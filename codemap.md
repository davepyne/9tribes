## NEW SYSTEMS

### captureSystem.ts — Contract
- **PURPOSE**: Slaver unit mechanic — capture enemy units instead of killing them
- **INPUT**: attacker (Unit), defender (Unit), RulesRegistry, optional greedyAbility (SignatureAbilityParams), optional rngState
- **OUTPUT**: `{ captured: boolean; state: GameState }`
- **KEY EXPORTS**: `attemptCapture`, `attemptNonCombatCapture`, `hasCaptureAbility`, `getCaptureParams`, `getCaptureCooldownRemaining`
- **SIDE EFFECTS**: On success — changes defender's factionId, resets HP to hpFraction of max, resets morale/veteranLevel, updates faction.unitIds, writes history entries (`captured`, `capture_attempt`). On failure — destroys unit normally.
- **INVARIANTS**: Cooldown tracked via unit.history entries (type=`capture_attempt`), not a dedicated counter. Greedy passive capture (SignatureAbility) falls back if no component-based capture.
- **CALLERS**: (new, not yet wired to callers based on import graph)
- **NOTE**: `attemptNonCombatCapture` is for Slave Galley / pirate units — converts without combat, costs attacksRemaining.

### transportSystem.ts — Contract
- **PURPOSE**: Naval transport of land units (e.g., Galley carrying infantry)
- **INPUT**: GameState, UnitId(s), RulesRegistry, TransportMap (externally managed)
- **OUTPUT**: `{ state: GameState; transportMap: TransportMap }`
- **KEY EXPORTS**: `boardTransport`, `disembarkUnit`, `canBoardTransport`, `canDisembark`, `destroyTransport`, `updateEmbarkedPositions`, `isUnitEmbarked`, `getValidDisembarkHexes`
- **SIDE EFFECTS**: Board — adds unitId to TransportMap. Disembark — moves unit to target hex, spends moves for both unit and transport. Destroy — cascading: removes transport + all embarked units.
- **INVARIANTS**: TransportMap is NOT part of GameState — caller must manage it. Embarked units stay in state.units but track transport position. Water terrains: `coast`, `river`, `ocean`. Disembark hex must be adjacent land, unoccupied.
- **CALLERS**: (new, not yet wired to callers based on import graph)

### fogSystem.ts — Contract
- **PURPOSE**: Fog of War — manages per-faction visibility (hidden/explored/visible) and last-seen snapshots
- **INPUT**: GameState, FactionId, RulesRegistry
- **OUTPUT**: FactionFogState { hexVisibility: Map, lastSeen: Map }
- **KEY EXPORTS**: `calculateVisibility`, `getLastSeenEnemyUnits`, `getVisibleHexes`, `getExploredHexes`, `canSeeUnit`, `initFogState`, `resetFogState`, `mergeFogState`, `serializeFogState`, `deserializeFogState`
- **SIDE EFFECTS**: Pure computation — no GameState mutation. Caller must store FactionFogState externally.
- **INVARIANTS**: Visibility radii: units=3, cities=3, villages=2. Mounted/scout units get +1. Terrain blocks: mountains block LOS. Explored hexes show terrain but not units.
- **CALLERS**: web GameSession, worldViewModel (UI rendering)

### healingSystem.ts — Contract
- **PURPOSE**: Per-turn unit healing based on location, faction identity, synergies, and capability doctrines
- **INPUT**: GameState, factionId, RulesRegistry
- **OUTPUT**: Updated GameState with healed HP/morale
- **KEY EXPORTS**: `applyHealingForFaction`
- **SIDE EFFECTS**: Mutates unit HP/morale in GameState
- **INVARIANTS**: Healing rates: owned territory 10%, city garrison 50%, village 50%, field 5%. Faction identity bonus via `getHealingBonus()`. Synergies via `applyHealingSynergies()`. Nature aura via `signatureAbilitySystem.getNatureHealingAura()`.
- **CALLERS**: (new, likely warEcologySimulation turn processing)

### learnByKillSystem.ts — Contract
- **PURPOSE**: Units learn enemy faction ability domains by killing enemies — part of "Learn by Killing + Sacrifice to Codify" mechanic
- **INPUT**: killer (Unit), victim (Unit), victim factionId, RNGState
- **OUTPUT**: LearnFromKillResult { unit, learned, domainId?, fromFactionId? }
- **KEY EXPORTS**: `attemptLearnFromKill`, `getVeteranLevelIndex`, `MAX_LEARNED_ABILITIES`
- **SIDE EFFECTS**: On success — adds learned ability to killer's `learnedAbilities[]`, emits TraceAbilityLearnedEvent
- **INVARIANTS**: Base learn chance 25%, +5% per veteran level (green→elite). Max 3 learned abilities per unit. Cannot learn own faction's domains.
- **CALLERS**: warEcologySimulation (combat kill flow)

### sacrificeSystem.ts — Contract
- **PURPOSE**: Units sacrifice learned abilities at home city to permanently add to faction's research tree — the "Codify" half of Learn+Codify
- **INPUT**: GameState, unitId, FactionId, RulesRegistry
- **OUTPUT**: { state: GameState, sacrificed: boolean, researchNodeId?: ResearchNodeId }
- **KEY EXPORTS**: `canSacrifice`, `performSacrifice`, `getSacrificeableAbilities`
- **SIDE EFFECTS**: On success — removes learned ability from unit, creates research node in faction tree, emits TraceUnitSacrificedEvent, writes history entry
- **INVARIANTS**: Unit must be on home city hex, home city not besieged, unit has ≥1 learned ability. Creates a new research node at the domain's root if no existing node for that domain.
- **CALLERS**: warEcologySimulation (activation loop)

### villageCaptureSystem.ts — Contract
- **PURPOSE**: Pirate's Greedy trait — capture enemy villages instead of destroying them
- **INPUT**: GameState, captor faction, target VillageId, RulesRegistry, VillageCaptureCooldownMap
- **OUTPUT**: `{ state: GameState; cooldownMap: VillageCaptureCooldownMap; success: boolean }`
- **KEY EXPORTS**: `hasGreedyTrait`, `attemptVillageCapture`, `canCaptureVillage`, `getVillageCaptureCooldownRemaining`
- **SIDE EFFECTS**: Converts village ownership to capturing faction. Records cooldown to prevent immediate recapture. Calls `destroyVillage` → `syncFactionSettlementIds`.
- **INVARIANTS**: Cooldown tracked in separate VillageCaptureCooldownMap (not GameState). Requires captor unit adjacent to village.
- **CALLERS**: (new, not yet wired to callers based on import graph)

## MODIFIED SYSTEMS (significant changes)

### combatSystem.ts — Contract
- **INPUT**: attacker (Unit), defender (Unit), terrain, registry, rngState
- **OUTPUT**: CombatResult
- **SIDE EFFECTS**: Mutates unit HP, generates combat signals, morale effects
- **INVARIANTS**: Defender always gets counter-attack unless killed
- **CALLERS**: opportunityAttackSystem (direct), warEcologySimulation (via combat signal flow)

### movementSystem.ts — Contract
- **INPUT**: GameState, UnitId, path, registry
- **OUTPUT**: Updated GameState with new unit position
- **SIDE EFFECTS**: Moves unit, triggers opportunity attacks, applies faction identity bonuses (Desert Swarm etc.), capability doctrine effects, synergy bonuses
- **INVARIANTS**: ZoC checked before movement, opportunity attacks interrupt movement
- **CALLERS**: warEcologySimulation (AI activation), web GameSession (player input), worldViewModel (UI)

### signatureAbilitySystem.ts — Contract
- **INPUT**: GameState, unitId, ability domain, registry, ability-domains.json
- **OUTPUT**: GameState with ability effects applied
- **SIDE EFFECTS**: Faction-specific powers (Frost Nova, Desert Swarm, etc.)
- **CALLERS**: warEcologySimulation (activation loop)

### territorySystem.ts — Contract
- **INPUT**: GameState, factionId, registry
- **OUTPUT**: Territory control data (claimed hexes, supply lines)
- **CALLERS**: warEcologySimulation, economySystem, siegeSystem, web worldViewModel

### warEcologySimulation.ts — Contract (CENTRAL ORCHESTRATOR)
- **PURPOSE**: Main simulation loop — activates all units per turn
- **INPUT**: GameState, RulesRegistry, turn config
- **OUTPUT**: Fully advanced GameState after one complete turn
- **DEPENDENCIES**: movementSystem, combatSystem, moraleSystem, xpSystem, veterancySystem, productionSystem, economySystem, villageSystem, territorySystem, zocSystem, synergyEngine, researchSystem, knowledgeSystem, factionStrategy, aiProductionStrategy, aiResearchStrategy, capabilityDoctrine, and content JSON files
- **CALLERS**: replay system, scripts
- **NOTE**: This is the "god function" — 31 import dependencies. Changes here cascade everywhere.

### siegeSystem.ts — Contract
- **INPUT**: GameState, attacker/defender, registry
- **OUTPUT**: Siege result, wall degradation
- **SIDE EFFECTS**: Degrades walls, handles city capture mechanics
- **CALLERS**: territorySystem (indirect), warEcologySimulation (activation)

### moraleSystem.ts — Contract
- **INPUT**: GameState, unitId, event type, registry
- **OUTPUT**: Updated morale value
- **CALLERS**: combatSystem, warEcologySimulation

### productionSystem.ts — Contract
- **INPUT**: GameState, factionId, registry
- **OUTPUT**: Production queue results, new units/city projects
- **CALLERS**: warEcologySimulation, aiProductionStrategy

### strategicAi.ts — Contract
- **INPUT**: GameState, factionId, registry
- **OUTPUT**: AI decisions (production, research, unit movement priorities)
- **CALLERS**: warEcologySimulation (AI faction turns)

## Frontend (web/src/)

- `web/src/game/controller/GameSession.ts` — Main game controller, calls movementSystem
- `web/src/game/view-model/worldViewModel.ts` — UI state model, calls movementSystem, territorySystem. Contains `getSpriteKeyForUnit()` — 3-tier lookup (sourceRecipeId → special chassis → faction+chassis) mapping all 39 playtest sprites.
- `web/src/game/phaser/assets/keys.ts` — Texture key registry + `getUnitTextureSpec()`. Uses data-driven `PLAYTEST_SPRITE_KEYS` lookup for all 39 faction sprites.
- `web/src/game/phaser/assets/assetManifest.ts` — Asset loading manifest. 39 playtest unit images loaded from `/assets/playtest-units/`.
- `web/src/game/phaser/systems/UnitRenderer.ts` — Renders units on the hex map using texture specs from keys.ts
- `web/src/game/phaser/systems/FogRenderer.ts` — Fog of war rendering (modified this cycle)
- `web/src/game/types/playState.ts` — Play state types
- Vite + TypeScript, separate build from `src/`

### Sprite System
- All 9 factions have unique per-unit sprites (39 total) in `web/public/assets/playtest-units/`
- Naming convention: `{faction}_{unit}.png` (e.g., `jungle_serpent.png`, `frost_polar_bear.png`)
- Display size: 48×64, yOffset: 8 (consistent across all playtest sprites)
- Starting units mapped by faction+chassisId, hybrid/signature units by sourceRecipeId or special chassisId
